# Problems.md — Diaphora MCP (Статус: Решено ✅)

В данном файле перечислены проблемы, обнаруженные в процессе разработки и тестирования, а также методы их исправления. Все 11 проблем успешно устранены.

---

### 1. Headless export (idat.exe) падает с error code 4
* **Симптом:** `idat.exe` завершается с ошибкой инициализации библиотеки, если на диске остались файлы блокировок от предыдущего падения.
* **Решение:** ✅ **Решено**. Перед каждым запуском экспорта вызывается автоматическая функция `_clean_stale_locks()`, очищающая `.id0`, `.id1` ... лок-файлы.

### 2. Crash-файл остаётся даже при успешном экспорте
* **Симптом:** Временный файл `-crash` не удаляется Diaphora, из-за чего экспорт ошибочно помечается как упавший.
* **Решение:** ✅ **Решено**. Теперь MCP-сервер проверяет реальное наличие функций и валидность SQLite базы, не полагаясь исключительно на файл `-crash`.

### 3. SQLite WAL не контроль-поинтится
* **Симптом:** При записи в режиме WAL данные могут оставаться в журнале, и база выглядит пустой (77KB) до применения чекпоинта.
* **Решение:** ✅ **Решено**. Во все проверки баз добавлено принудительное выполнение `PRAGMA wal_checkpoint(PASSIVE)`.

### 4. `true_name` column отсутствует в некоторых схемах
* **Симптом:** Ошибка `no such column: true_name` при переносе метаданных на старых версиях Diaphora.
* **Решение:** ✅ **Решено**. Добавлен автоматический try/except fallback на стандартную колонку `name`.

### 5. Медленный экспорт (1.2GB .i64)
* **Симптом:** На огромных базах процесс висит без прогресса до 10 минут.
* **Решение:** ✅ **Решено**. Реализован сторожевой таймер (Watchdog) по дисковой активности файлов SQLite/WAL и гибко настраиваемые таймауты.

### 6. Нет мониторинга прогресса экспорта
* **Симптом:** Запуск subprocess блокирует поток, скрывая прогресс.
* **Решение:** ✅ **Решено**. Добавлено фоновое логирование размера WAL-файла и промежуточных шагов экспорта.

### 7. Windows file locking мешает cleanup
* **Симптом:** Заблокированные файлы `.id0`, `.sqlite-wal` вызывают PermissionError 32 при попытке удаления.
* **Решение:** ✅ **Решено**. Написана утилита `force_delete_file()`, которая автоматически принудительно убивает процессы `idat.exe` через `taskkill` при блокировке и завершает удаление.

### 8. Неинформативные логи при падении
* **Симптом:** stdout/stderr пустые, ошибки пишутся только в `ida.log`.
* **Решение:** ✅ **Решено**. Улучшен захват вывода subprocess, а также добавлен tail-логгер для вывода последних 2K текста stdout/stderr в ответ MCP.

### 9. `batch_export_and_diff` не удаляет промежуточные файлы
* **Симптом:** Временные базы засоряют диск.
* **Решение:** ✅ **Решено**. Добавлен параметр `cleanup: bool = True` (по умолчанию `True`) для автоматической очистки всех промежуточных файлов в блоке `finally`.

### 10. `_read_results` в diff.py обрезает results до 500
* **Симптом:** Результаты обрезались до 500 строк без возможности настройки.
* **Решение:** ✅ **Решено**. Добавлены параметры `limit` и `unmatched_limit` с флагами `truncated` и `unmatched_truncated` для точного контроля размера выдачи.

### 11. idat.exe не завершается после успешного экспорта
* **Симптом:** Процесс `idat.exe` зависает в памяти после завершения экспорта и потребляет до 3.5GB RAM.
* **Решение:** ✅ **Решено**. В скрипт-обертку `_diaphora_headless.py` добавлен принудительный вызов `idaapi.qexit(0)` для чистого выхода.

### Дополнительно реализовано:
12. **GUI-интеграция (XML-RPC Мост):** Создан плагин `diaphora_gui_listener.py` для автоматического экспорта прямо из запущенной GUI IDA Pro (на порту `28652`) без конфликтов блокировки.
13. **Лимит рекурсии Python:** Лимит рекурсии поднят до `100000` (`sys.setrecursionlimit`), что предотвращает падения `maximum recursion depth exceeded` на базах с 150k+ функциями.
